# v0

## JSON Schema spec correctness fixes

> Investigated in
> [`../plans/plan_json_schema_correctness.md`](../plans/plan_json_schema_correctness.md).

Four keyword implementations contradict the JSON Schema Validation 2020-12
spec and must be fixed before the library can be considered correct:

- `pattern` is incorrectly anchored (spec says patterns are not implicitly anchored)
- `integer` type rejects numbers with zero fractional part (e.g. `1.0`)
- `type` only accepts a string value; array-of-types form is silently ignored
- `uri` and `urn` format validators are swapped

## JSON Schema: wire existing validators into the rule layer

> Investigated in
> [`../plans/plan_json_schema_wire_validators.md`](../plans/plan_json_schema_wire_validators.md).

Six keywords have working `Validator` implementations in `validation.dart` but
are not reachable via `SchemaParser` or `JsonSchemaValidator` because they have
no `SchemaRule` subtype and no parser branch:

`const`, `multipleOf`, `uniqueItems`, `minProperties`, `maxProperties`,
`dependentRequired`

## JSON Schema: missing keywords

> Investigated in
> [`../plans/plan_json_schema_missing_keywords.md`](../plans/plan_json_schema_missing_keywords.md).

Structural keywords that are absent entirely and whose omission can cause
invalid documents to silently pass validation:

- `contains` / `minContains` / `maxContains`
- `prefixItems` (and correct 2020-12 semantics for `items` with a prefix)
- `patternProperties`
- `additionalProperties` as a sub-schema (currently only `false` is handled)

## JSON Schema: standard format strings

> Investigated in
> [`../plans/plan_json_schema_formats.md`](../plans/plan_json_schema_formats.md).

Eleven format strings defined by the spec (§7.3) are not registered in
`StringFormatValidator`. Schemas using these formats silently produce no
violations today:

`idn-email`, `hostname`, `idn-hostname`, `ipv4`, `ipv6`, `uri-reference`,
`iri`, `iri-reference`, `uri-template`, `json-pointer`, `relative-json-pointer`
